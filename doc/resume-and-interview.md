# WorkBrain 简历与面试讲解稿

## 简历项目描述

### 项目名称

WorkBrain — 企业知识库与 IT 服务申请助手

### 一句话版本

基于 FastAPI、React、PostgreSQL/pgvector、Celery 和原生 Tool Calling 构建多租户企业知识库与 IT 服务申请助手，实现带引用 RAG、RBAC、异步文档处理、Agent 两阶段确认、审批审计和 Docker 化部署。

### 三条简历版本

- 设计组织、成员和 `member/approver/admin` RBAC 模型，将 `organization_id` 隔离应用到知识库、文档、RAG、服务目录、申请、日志和 Agent Trace，覆盖跨组织越权与自审批防护测试。
- 使用 Celery + Redis 构建文档解析、切块和向量化任务，支持原子领取、重试、幂等、状态查询；基于 PostgreSQL/pgvector 实现语义与词法联合检索、引用来源和资料不足拒答。
- 使用原生 Tool Calling 实现企业服务 Agent，将创建申请拆为“生成确认内容”和“用户提交一次性确认令牌”两阶段，并通过 React 前端、Alembic、Docker Compose、健康检查、生产冒烟及 300 级别后端测试完成交付。

## 30 秒介绍

> WorkBrain 是我从零实现的企业知识库与 IT 服务申请助手。它支持多组织 RBAC、异步文档处理、带引用且会拒答的 RAG，以及查询服务、补全信息并在用户确认后创建申请的 Agent。项目最重要的设计是把 LLM 限制在后端注册工具和组织权限之内，并为写操作增加一次性确认、幂等、审计和自动化测试。前端、Worker、数据库、网关和迁移可以通过 Docker Compose 完整运行。

## 两分钟介绍

> 这个项目解决两个相关问题：员工如何可信地查询企业知识，以及如何快速完成 IT 服务申请。
>
> 数据层以组织为租户边界。JWT 识别用户，`X-Organization-ID` 选择工作空间，后端再查询有效成员关系和角色。知识库、文档、RAG 日志、服务项目和申请查询都会同时限制组织编号。
>
> 文档上传后不会在 HTTP 请求里同步解析，而是创建后台任务，由 Celery Worker 完成解析、切块和 Embedding。任务支持原子领取、重试和幂等。只有发布后的分块才能进入 RAG，检索结果还要经过词法匹配和最低分数过滤；证据不足时直接拒答，正常回答附带来源编号。
>
> 服务 Agent 使用原生 Tool Calling 查询当前组织目录和用户申请。涉及创建申请时，模型只能生成待确认内容，不能直接写数据库。用户提交一次性确认令牌后，后端重新验证组织、用户和服务状态，再幂等创建 pending 申请。approver/admin 可以批准或填写原因拒绝，申请人不能审批自己，所有状态变化都有审计记录。
>
> 最后我补了 React 前端、Alembic、Docker Compose、Caddy HTTPS 配置、健康检查、RAG 评测集和生产冒烟脚本，使它成为可以运行和讲解的完整项目。

## 核心设计问题

### 为什么 JWT 之后还需要组织上下文

JWT 只能回答“当前是谁”，不能回答“当前代表哪个组织操作”。`X-Organization-ID` 显式选择租户，后端根据用户和组织查询有效 Membership，再根据角色执行策略。每个资源查询仍然带上 `organization_id`，防止只校验入口却在数据库层泄漏数据。

### 为什么文档处理使用 Celery

解析、切块和向量化耗时且可能依赖外部 API。如果放在上传请求里，容易超时，也无法可靠重试。异步任务让 API 快速返回 `202`，Worker 独立消费任务，并记录 queued、running、succeeded、failed 等状态。

### 如何防止 Worker 重复处理

任务领取使用条件更新实现原子状态流转；重复消息只能有一个 Worker 成功领取。文档版本和任务状态共同形成幂等边界，失败重试使用退避策略，结束状态不会再次执行。

### RAG 如何减少幻觉

1. 只检索当前组织和知识库中已发布的文档。
2. 使用向量相似度寻找候选分块。
3. 增加词法匹配数量和相关度过滤。
4. 使用最低综合分数阈值。
5. 没有证据时不调用生成模型回答事实。
6. 上下文带 `[S1]` 编号，响应返回对应文档和分块。

这些措施降低风险，但不能保证模型永远正确，因此不能把 RAG 描述成事实数据库。

### 为什么 Agent 使用两阶段确认

自然语言可能有歧义，模型输出也不等于用户授权。第一阶段只解析意图、补全字段并生成确认内容；第二阶段由用户明确提交确认令牌。后端在第二阶段重新验证用户、组织、服务启用状态和令牌消费状态，模型无法伪造确认。

### 如何实现确认幂等

数据库只保存确认令牌哈希，不保存原始令牌。确认记录绑定用户、组织、服务和申请内容，并记录过期时间、消费时间及生成的申请编号。重复提交已经消费的有效令牌时返回同一结果，不会创建第二条申请。

### 为什么使用原生 Tool Calling

当前 Agent 的流程短、工具数量有限，主要难点是业务权限和确认边界。原生 Tool Calling 能直接观察模型请求、工具参数、执行结果和 Trace，依赖更少，也更适合展示底层原理。

### 为什么没有使用 LangGraph

当前流程没有跨小时暂停、复杂分支、checkpoint 恢复或多个专家 Agent handoff。引入 LangGraph 会增加状态图和持久化复杂度，但不会直接解决当前核心问题。项目保留了方案对比文档；当流程出现长时间运行、多节点恢复或复杂人工介入时再重新评估。

### MCP 在项目中的位置

项目实现了一个只读 stdio MCP Demo，只返回固定组织的启用服务目录，用于理解工具协议和安全注解。它没有生产身份认证和写操作，因此没有被包装成生产 MCP 服务。

### 如何证明跨组织隔离

不能只展示前端切换。自动化测试会：

- 创建两个组织和不同成员关系。
- 使用组织 A 的 JWT 与组织 B 的资源编号发起请求。
- 验证返回 403/404。
- 检查 RAG、申请、目录、任务和日志没有跨组织结果。

### 生产部署做了什么

- PostgreSQL、Redis AOF、上传文件和 Caddy 数据使用持久卷。
- Alembic 迁移作为一次性服务先于 API 和 Worker。
- API 与 Worker 使用非 root 用户。
- 生产编排只暴露 Caddy 的 80/443。
- Nginx 托管 SPA，并将 `/api` 代理到 FastAPI。
- liveness 检查进程，readiness 检查数据库和 Broker。
- 生产冒烟脚本走通写入、审批、审计和后台任务。

## STAR 讲解模板

### Situation

传统聊天 Demo 缺少企业权限、数据隔离和高风险操作确认，无法直接用于知识库和 IT 申请场景。

### Task

构建一个能部署、能测试、带前端，并能完整演示“知识查询 + 服务申请 + 审批审计”的 MVP。

### Action

- 先用组织 Membership 和 RBAC 建立租户边界。
- 将文档处理拆成 Celery 异步任务。
- 使用 pgvector、词法过滤、阈值、引用和拒答建立可信 RAG。
- 将 Agent 写操作改造成两阶段确认。
- 增加申请状态机、审计事件和跨组织测试。
- 使用 React、Docker Compose、Caddy、健康检查和冒烟脚本完成交付。

### Result

形成从注册、组织、知识库、文档、RAG、Agent 申请到审批审计的完整闭环，并具备 300 级别后端测试、47 个前端测试、迁移检查和可重复生产验收。

## 面试追问速答

### 如果 Embedding API 失败怎么办

Worker 将任务标记为失败并保存安全错误信息；符合重试条件时按退避策略重试。失败不会把未完成文档发布到 RAG。

### 如果确认后服务被停用怎么办

确认阶段重新查询当前组织内仍启用的服务项目，失效时拒绝创建，不能依赖第一阶段的旧结果。

### 为什么拒绝理由是必填

拒绝属于业务决定，需要给申请人可解释结果，也需要在审计中留下依据。批准可以没有原因，拒绝必须提供非空原因。

### 为什么没有复杂多级审批

MVP 聚焦单级审批闭环。多级审批会引入流程定义、步骤实例、转派、超时和补偿，需要独立工作流设计，不应在没有真实需求时提前加入。

### 下一步如何扩展

优先考虑对象存储、密码重置、线上监控、备份恢复和真实企业 RAG 评测集。只有出现明确长流程需求时才考虑 LangGraph 或 BPM 引擎。

## 不能声称的内容

面试和简历中不能声称：

- 已经服务真实付费客户，除非确实发生。
- 已经部署 Kubernetes、多区域高可用或自动扩缩容。
- RAG 可以完全消除幻觉。
- MCP Demo 是生产级 MCP 服务。
- 当前评测集能代表所有企业数据。
- Docker Compose 等于完整云原生平台。

准确表达比堆砌技术名词更有说服力。可以说“具备生产化单机部署制品并完成本地生产式验收”，不要在没有公网证据时说“已经正式上线”。
